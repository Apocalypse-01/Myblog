
基于Spring Boot与Vue 3的个人博客系统设计与实现

摘  要

随着互联网技术的飞速发展，个人博客作为一种重要的信息传播和知识分享平台，在学术交流、技术分享和个人品牌建设中扮演着越来越重要的角色。传统的博客平台虽然功能完善，但存在定制化程度低、数据可控性差、广告干扰等问题。针对这些问题，本文设计并实现了一个基于Spring Boot与Vue 3的个人博客系统。

本系统采用当前业界主流的前后端分离架构。前端使用Vue 3框架结合TypeScript语言和Vite构建工具，配合Pinia状态管理、Vue Router路由管理和Element Plus UI组件库，构建了响应式用户界面；后端基于Spring Boot 3框架提供RESTful API服务，使用MyBatis作为持久层框架，MySQL作为数据库管理系统，采用JWT（JSON Web Token）实现无状态身份认证，BCrypt算法进行密码加密，并通过Knife4j自动生成API文档。

系统实现了用户管理（注册、登录、个人信息管理、头像上传）、文章管理（发布、编辑、列表展示、详情阅读、搜索、多格式下载）和社交互动（点赞、收藏、评论）等核心功能。在UI设计方面，系统采用了毛玻璃视觉效果和三栏响应式布局，并通过CSS Flexbox实现了不同屏幕尺寸的适配。同时，系统在安全性方面采取了多层防护措施，包括JWT认证拦截、BCrypt密码加密、文件上传安全校验、防XSS攻击和CORS跨域配置等。

本文按照软件工程的规范流程，详细阐述了系统的需求分析、技术选型、架构设计、数据库设计、功能模块实现、系统测试等完整开发过程。最终实现的个人博客系统功能完善、界面美观、安全可靠，具有良好的可维护性和可扩展性。

关键词：个人博客；Spring Boot；Vue 3；前后端分离；RESTful API；JWT认证

第1章  绪论

1.1 项目背景与意义

随着互联网技术的飞速发展，个人博客作为一种重要的信息传播和知识分享平台，在学术交流、技术分享和个人品牌建设中扮演着越来越重要的角色。在Web 2.0时代，博客成为个人在互联网上表达观点、分享知识、记录成长的重要载体。根据中国互联网络信息中心（CNNIC）的统计数据，截至2025年，中国网民规模已超过11亿，互联网普及率持续提升，这为个人博客系统的发展提供了广阔的用户基础。

传统的博客平台（如CSDN、博客园、简书等）虽然功能完善，但存在以下显著问题：（1）定制化程度低，用户无法自由控制页面的布局和样式；（2）数据可控性差，用户的文章数据存储在第三方平台，存在数据丢失或平台关闭的风险；（3）广告干扰严重影响阅读体验；（4）功能同质化严重，缺乏个性化特色。因此，开发一个功能完善、界面美观、安全可靠且可自主掌控的个人博客系统具有重要的实际意义和应用价值。

本系统采用当前业界主流的前后端分离架构。前端使用Vue 3框架构建响应式用户界面，结合TypeScript确保代码的类型安全性，Element Plus组件库提供高质量的UI组件支持；后端基于Spring Boot 3框架提供RESTful API服务，MyBatis作为持久层框架实现灵活的数据访问，MySQL作为数据库管理系统进行数据持久化存储。系统实现了用户管理、文章发布与管理、社交互动（点赞、收藏、评论）、全文搜索、文章多格式下载等核心功能，并支持响应式布局以适配多终端访问。在设计风格上，系统采用了毛玻璃（Glassmorphism）视觉效果，为用户提供现代化的审美体验。

本项目的开发不仅有助于解决传统博客平台的不足，也为学习现代Web开发技术栈（Vue 3 + Spring Boot 3）提供了一个完整的实践案例。通过本系统的设计与实现，可以深入理解前后端分离架构的设计理念、RESTful API的设计规范、数据库设计方法以及Web安全防护措施等关键知识点，具有较强的理论意义和实践价值。

1.2 国内外研究现状

博客系统的研究与发展可以追溯到20世纪90年代末期。1999年，Pyra Labs推出了Blogger平台，被认为是现代博客系统的开端。此后，WordPress（2003年发布）迅速崛起，凭借其开源特性、丰富的插件生态和灵活的模板系统，成为全球最受欢迎的博客和内容管理系统（CMS），目前全球约43%的网站由WordPress驱动。

从技术演进的角度来看，博客系统的发展呈现出以下趋势：（1）前后端分离成为主流架构模式，前端框架从jQuery演进到React、Vue等现代框架；（2）静态站点生成器（SSG）和服务端渲染（SSR）技术提升了性能和SEO效果；（3）Markdown成为主流的写作格式；（4）响应式设计和移动端优化成为标配；（5）安全性受到越来越多的重视，JWT认证、HTTPS加密、内容安全策略（CSP）等措施被广泛采用。

本系统在设计时充分吸收了上述研究的成果，结合Vue 3和Spring Boot 3的最新技术，构建了一个功能全面、架构清晰、用户体验良好的个人博客系统。

1.3 系统目标与功能定位

基于对现有博客系统的分析和用户需求调研，本系统的核心目标包括以下六个方面：

（1）提供功能完善的个人博客内容管理系统，支持文章的增删改查全生命周期管理，包括文章的发布、编辑、删除、分类和列表展示等功能。

（2）实现用户注册、登录、个人信息管理等用户中心功能，提供安全可靠的身份认证机制，保护用户账号和数据安全。

（3）构建社交互动体系，包括文章点赞、收藏和评论功能，增强用户之间的互动性和社区氛围。

（4）支持文章全文搜索和多种格式下载（Markdown、TXT、EPUB），满足用户在不同场景下的阅读和使用需求。

（5）采用现代化的UI设计理念，运用毛玻璃视觉效果和响应式布局技术，提供美观、流畅的用户体验，适配桌面端和移动端设备。

（6）确保系统的安全性和数据可靠性，通过多层安全防护措施保护用户数据，包括密码加密存储、JWT身份认证、防XSS攻击、文件上传安全校验等。


第2章  相关技术介绍

2.1 前端技术

2.1.1 Vue 3框架

Vue 3是尤雨溪团队开发的渐进式JavaScript框架的最新主版本（本系统使用v3.5.13），于2020年9月正式发布。相比Vue 2，Vue 3带来了以下显著改进：

（1）Composition API（组合式API）：提供更灵活的逻辑复用和代码组织方式。开发者可以将相关的响应式状态、计算属性和方法组织在一起，形成可复用的组合函数（Composables）。本系统广泛使用了Composition API的setup语法糖（`<script setup>`），结合ref和reactive实现响应式数据绑定。

（2）更好的TypeScript支持：Vue 3的源代码使用TypeScript重写，提供了一流的TypeScript类型推导支持，使得在开发过程中能够获得更完善的IDE智能提示和类型检查。

（3）更小的包体积：通过Tree-shaking（摇树优化）实现按需打包，未使用的功能在构建时会被自动移除，减少了最终打包体积。

（4）更快的渲染性能：基于ES2015 Proxy实现响应式系统，替代了Vue 2中基于Object.defineProperty的实现。Proxy可以拦截更多类型的操作（如属性的添加和删除、数组索引和length的修改等），同时提升了响应式系统的性能。


2.2 后端技术

2.2.1 Spring Boot 3框架

Spring Boot 3（本系统使用v3.3.10）是基于Spring Framework 6的新一代Java开发框架，要求Java 17及以上的运行环境。Spring Boot的核心设计理念是"约定优于配置"，通过自动配置和Starter依赖极大简化了Spring应用的搭建和开发过程。

2.2.2 MyBatis持久层框架

MyBatis（本系统使用v3.5.10，配合spring-boot-starter v3.0.3）是一款优秀的Java持久层框架，它通过XML或注解的方式将Java对象与SQL语句进行映射，支持自定义SQL、存储过程和高级映射。


2.2.3 MySQL数据库

MySQL是由Oracle公司维护的开源关系型数据库管理系统，本系统使用MySQL 8.0版本。MySQL的选择基于以下理由：（1）开源免费，拥有活跃的全球社区支持；（2）成熟稳定，经过大量生产环境的验证；（3）InnoDB存储引擎提供完整的事务支持（ACID特性）和行级锁定机制；（4）UTF-8字符集（utf8mb4）支持，能够满足多语言内容的存储需求。

3.1.1 技术可行性

本系统所采用的技术栈均为当前业界成熟且广泛应用的技术。前端方面，Vue 3框架拥有庞大的用户社区和完善的生态系统，Element Plus提供了丰富的UI组件，Vite作为新一代构建工具已经过大量项目的验证。后端方面，Spring Boot 3是Java生态中最流行的微服务开发框架，MyBatis是成熟的持久层框架，MySQL是经过数十年考验的关系型数据库管理系统。开发团队具备相关技术的实践经验，开发工具（VS Code和IntelliJ IDEA）功能完善。综合评估，本系统在技术层面完全可行。

3.1.2 经济可行性

本系统的开发和部署成本较低。所有核心技术（Vue 3、Spring Boot、MySQL、MyBatis等）均为开源免费软件，无需支付商业许可费用。开发过程中使用的IDE和工具（VS Code社区版、IntelliJ IDEA社区版）同样提供免费的社区版本。部署方面，初期可以选择低成本的云服务器方案，系统资源需求不高，运行成本可控。因此，本系统在经济层面具有良好的可行性。

3.1.3 操作可行性

本系统面向互联网用户，UI设计遵循现代Web应用的设计规范，采用了直观的导航结构和友好的操作流程。文章编辑采用Markdown编辑器，技术用户能够快速上手；基础功能（注册、登录、浏览文章）操作简单，普通用户也可以轻松使用。系统支持响应式布局，能够在桌面端和移动端设备上提供一致的使用体验。整体而言，系统在操作层面具有良好的可行性。

3.2 功能需求分析

3.2.1 用户模块需求

用户模块是系统的基础模块，负责用户身份的管理。核心功能需求包括：

（1）用户注册：新用户通过填写用户名、邮箱、密码完成账号注册。系统需要进行前端表单验证（邮箱格式、密码长度、手机号格式）和后端数据校验（邮箱唯一性检查）。

（2）用户登录：已注册用户使用邮箱和密码进行身份认证。登录成功后系统生成JWT Token，前端存储该Token并维持登录状态。Token过期后需要自动或提示用户重新登录。

（3）个人信息管理：用户可以查看和编辑个人基本信息，包括用户名、邮箱、手机号、个性签名等。信息修改后需要实时更新。

（4）头像上传：用户可以上传自定义头像图片。系统需要对上传文件进行类型（仅限图片格式）和大小（不超过2MB）的限制，并对文件名进行安全处理。

3.2.2 文章模块需求

文章模块是系统的核心业务模块，承载了内容创作和管理的主体功能。核心功能需求包括：

（1）文章发布：登录用户可以使用Markdown编辑器编写文章，设置标题和分类标签（日常/技术/杂谈/其他）。系统需要支持草稿暂存功能，防止用户意外丢失编辑内容。

（2）文章编辑：文章作者可以修改已发布文章的内容、标题和标签，系统需要验证操作者是否具有编辑权限（仅文章作者可编辑）。

（3）文章列表展示：以卡片形式展示文章列表，支持按分类标签筛选，支持按点赞数或收藏数排序，支持分页浏览（每页10条）。每篇文章卡片显示标题、标签、作者名称和发布时间。

（4）文章详情阅读：展示文章完整内容，包括标题、作者信息、发布时间、正文内容和评论区。页面需要提供点赞、收藏、文章下载等操作入口。

（5）文章搜索：支持通过关键词搜索文章标题和内容。搜索结果需要支持分页展示。

（6）文章下载：支持将文章内容导出为Markdown（.md）、纯文本（.txt）和EPUB（.epub）三种格式文件，供用户离线阅读。

（7）文章删除：文章作者可以删除自己发布的文章。

3.2.3 互动模块需求

互动模块旨在增强用户之间的社交互动，提升社区的活跃度。核心功能需求包括：

（1）文章点赞：登录用户可以对文章进行点赞操作，再次点击可取消点赞。系统需要防止重复点赞，并实时更新文章的点赞数量。

（2）文章收藏：登录用户可以将文章加入个人收藏夹，再次点击可取消收藏。系统需要防止重复收藏，并实时更新文章的收藏数量。

（3）文章评论：登录用户可以发表评论，评论内容不能为空。用户可以查看某篇文章的所有评论，按时间顺序排列。评论的作者可以删除自己的评论，系统需要验证删除权限。

实体之间的关系说明如下：
- 一个用户(User)可以创建多篇文章(Article)，为一对多关系（1:N），通过Article表中的author_id外键关联；
- 一篇文章(Article)可以有多条评论(article_comment)，为一对多关系（1:N），通过article_comment表中的article_id外键关联；
- 一条评论(article_comment)属于一个用户(User)，为一对一关系（1:1）；
- 用户与文章之间存在多对多的点赞关系，通过article_like中间表实现，该表记录article_id和author_id的对应关系；
- 用户与文章之间存在多对多的收藏关系，通过article_favorite中间表实现，与点赞关系的设计模式一致。

4.2.3 数据库表结构设计

基于E-R图的设计，系统共创建了五张数据表，具体表结构如下：

（1）用户表(user)

用户表用于存储系统注册用户的基本信息，是系统的核心基础表。表4-1展示了用户表的详细字段定义。

表4-1 用户表(user)结构

| 字段名       | 类型            | 约束                      | 说明                |
|-------------|-----------------|---------------------------|---------------------|
| id          | INT             | PRIMARY KEY AUTO_INCREMENT | 用户唯一标识         |

（2）文章表(article)

文章表是系统的核心业务表，存储所有文章的内容和元数据。表4-2展示了文章表的详细字段定义。

| user_name   | VARCHAR(50)     |                           | 评论用户名     |
| content     | TEXT            | NOT NULL                   | 评论内容       |
| create_time | DATETIME        |                           | 评论时间       |
| change_time | DATETIME        |                           | 修改时间       |

（4）点赞表(article_like)

点赞表用于记录用户对文章的点赞关系，通过唯一约束防止重复点赞。表4-4展示了点赞表的详细字段定义。

表4-4 点赞表(article_like)结构

| 字段名       | 类型            | 约束                      | 说明           |
|-------------|-----------------|---------------------------|----------------|
| id          | INT             | PRIMARY KEY AUTO_INCREMENT | 记录唯一标识    |
| article_id  | INT             | NOT NULL                   | 文章ID         |
| author_id   | INT             | NOT NULL                   | 点赞用户ID     |
| create_time | DATETIME        |                           | 点赞时间       |
| change_time | DATETIME        |                           | 修改时间       |

点赞表设置了(article_id, author_id)的唯一约束组合，从数据库层面保证了同一用户对同一文章只能点赞一次，避免了并发情况下的重复点赞问题。

（5）收藏表(article_favorite)

收藏表的设计与点赞表完全对称，用于记录用户对文章的收藏关系。表4-5展示了收藏表的详细字段定义。

表4-5 收藏表(article_favorite)结构

| 字段名       | 类型            | 约束                      | 说明           |
|-------------|-----------------|---------------------------|----------------|
| id          | INT             | PRIMARY KEY AUTO_INCREMENT | 记录唯一标识    |
| article_id  | INT             | NOT NULL                   | 文章ID         |
| author_id   | INT             | NOT NULL                   | 收藏用户ID     |
| create_time | DATETIME        |                           | 收藏时间       |
| change_time | DATETIME        |                           | 修改时间       |

收藏表同样设置了(article_id, author_id)的唯一约束组合，防止同一用户重复收藏同一篇文章。
