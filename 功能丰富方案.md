# 个人博客系统功能丰富方案

---

## 一、当前系统详细信息

### 1.1 系统类型

基于Vue3与Spring Boot的前后端分离个人博客系统，采用B/S架构，支持单用户及多用户博客内容发布与互动。

### 1.2 核心功能

| 模块 | 功能 | 当前状态 |
|-----|------|---------|
| 用户管理 | 注册、登录、退出 | ✅ 已实现 |
| 文章管理 | 发布、列表、分类、分页、详情 | ✅ 已实现 |
| 互动功能 | 点赞、收藏、我的点赞、我的收藏 | ✅ 已实现 |
| 身份认证 | JWT Token生成与验证 | ✅ 已实现 |
| 文章编辑 | Markdown编辑器（md-editor-v3） | ✅ 已实现 |
| 个人中心 | 个人信息、我的文章 | ⚠️ 部分实现 |
| 评论功能 | 评论展示与发表 | ❌ 未完成（接口存在但未贯通） |
| 文章搜索 | 按标题/内容搜索 | ❌ 未实现 |
| 用户关注 | 关注作者 | ❌ 未实现 |
| 草稿箱 | 文章暂存 | ❌ 未实现 |
| 消息通知 | 互动消息提醒 | ❌ 未实现 |
| 图片上传 | 文章配图上传 | ❌ 未实现 |

### 1.3 目标用户群体

| 用户类型 | 特征描述 | 核心需求 |
|---------|---------|---------|
| 个人博主 | 希望拥有独立博客空间的技术爱好者/写作者 | 文章发布、内容管理、个人品牌展示 |
| 知识分享者 | 希望分享技术文章或学习笔记的开发者 | Markdown编辑、代码高亮、分类标签 |
| 阅读用户 | 浏览和收藏优质内容的读者 | 文章浏览、收藏、点赞、搜索 |
| 潜在用户 | 对博客系统感兴趣但尚未注册的访客 | 快速注册、内容预览 |

## 三、功能丰富建议

### 3.1 第一阶段：核心功能补全（优先级：高）

#### 3.1.1 文章搜索功能

**功能描述**：
支持按文章标题、内容、作者名进行关键词搜索，提供快速定位文章的能力。

**用户需求**：
随着文章数量增长，用户需要快速找到目标文章，当前仅靠分类和翻页效率低下。

**技术可行性**：⭐⭐⭐⭐⭐（非常高）

**实现路径**：

| 层次 | 实现内容 |
|-----|---------|
| 数据库 | 在article表的article_title和article_content字段上添加FULLTEXT索引 |
| Mapper层 | 新增searchByKeyword方法，使用MATCH...AGAINST进行全文检索 |
| Service层 | 新增searchArticle方法，处理搜索逻辑和分页 |
| Controller层 | 新增GET /article/search接口，接收keyword和分页参数 |
| 前端API | 新增searchArticle方法 |
| 前端组件 | 在文章列表页顶部添加搜索输入框，支持实时搜索 |

**数据库变更**：
```
新增索引：ALTER TABLE article ADD FULLTEXT INDEX ft_search(article_title, article_content)
```

**预期效益**：
- 用户查找文章效率提升80%以上
- 降低用户因找不到内容而流失的概率

**资源需求**：
- 开发工时：2-3天
- 无额外技术依赖

| Entity层 | 新增Comment实体类 |
| DTO/VO层 | 新增CommentDTO、CommentVO |
| Mapper层 | 新增CommentMapper，实现insertComment、getCommentsByArticleId、deleteComment |
| Service层 | 新增CommentService及实现类 |
| Controller层 | 新增POST /comment/add、GET /comment/list、DELETE /comment/delete |

**预期效益**：

#### 3.1.3 密码加密存储

**功能描述**：
使用BCrypt算法对用户密码进行加密存储，替代当前的明文存储方式。

**用户需求**：
当前密码明文存储存在严重安全隐患，一旦数据库泄露将直接暴露用户密码。

**技术可行性**：⭐⭐⭐⭐⭐（非常高）

**实现路径**：

| 层次 | 实现内容 |
|-----|---------|
| 依赖 | 添加spring-boot-starter-security依赖（仅使用BCrypt部分） |
| 工具类 | 新增PasswordUtils，提供encrypt和verify方法 |
| Service层 | 注册时调用encrypt加密密码；登录时调用verify验证密码 |
| 数据库 | 对现有用户的明文密码进行一次性加密迁移 |

**预期效益**：
- 彻底解决密码明文存储的安全隐患
- 符合安全合规要求

**资源需求**：
- 开发工时：1-2天
- 需要数据迁移脚本

#### 3.2.2 草稿箱功能

**功能描述**：
支持文章保存为草稿，随时继续编辑，避免编辑中断导致内容丢失。

**用户需求**：
长文编辑过程中可能因意外关闭页面而丢失内容，草稿功能是写作类系统的标配。

**技术可行性**：⭐⭐⭐⭐（高）

**实现路径**：

| 层次 | 实现内容 |
|-----|---------|
| 数据库 | article表新增status字段（0=草稿，1=已发布） |
| Mapper层 | 新增saveDraft、getDraftsByAuthorId方法 |
| Service层 | 修改insertArticle逻辑，根据status区分草稿和发布 |
| Controller层 | 新增POST /article/saveDraft、GET /article/getDrafts |
| 前端组件 | release.vue中"保存草稿"按钮绑定实际功能；新增草稿列表页 |

**数据库变更**：
```
ALTER TABLE article ADD COLUMN status INT DEFAULT 1 COMMENT '0=草稿,1=已发布'
```

**预期效益**：
- 避免内容丢失，提升写作安全感
- 支持分次写作，提升写作效率

**资源需求**：
- 开发工时：2-3天
- 无额外技术依赖

#### 3.2.4 消息通知功能

**功能描述**：
当用户的文章被点赞、收藏、评论或获得新粉丝时，发送站内通知提醒。

**用户需求**：
用户希望及时了解自己文章的互动情况，当前无法获知任何反馈。

**技术可行性**：⭐⭐⭐（中）

**实现路径**：

| 层次 | 实现内容 |
|-----|---------|
| 数据库 | 新增notification表（id, user_id, type, content, related_id, is_read, create_time） |
| Entity层 | 新增Notification实体类 |
| Mapper层 | 新增NotificationMapper |
| Service层 | 新增NotificationService，在点赞/收藏/评论/关注时触发通知 |
| Controller层 | 新增GET /notification/list、PUT /notification/read |
| 前端组件 | 导航栏添加通知图标和未读数量角标；新增通知列表页面 |

**数据库变更**：
```
CREATE TABLE notification (
    id INT PRIMARY KEY AUTO_INCREMENT,
    user_id INT NOT NULL COMMENT '接收通知的用户ID',
    type VARCHAR(20) NOT NULL COMMENT '通知类型：like/favorite/comment/follow',
    content VARCHAR(500) COMMENT '通知内容',
    from_user_id INT COMMENT '触发通知的用户ID',
    related_id INT COMMENT '关联的文章/评论ID',
    is_read TINYINT DEFAULT 0 COMMENT '是否已读：0=未读,1=已读',
    create_time DATETIME COMMENT '创建时间',
    INDEX idx_user_read(user_id, is_read)
)
```

**预期效益**：
- 提升用户互动感知，增强社区参与感
- 提高用户回访率和活跃度

**资源需求**：
- 开发工时：4-5天
- 后期可扩展为WebSocket实时推送

## 五、与现有系统的兼容性分析

### 5.1 数据库兼容性

| 功能 | 数据库变更 | 兼容性影响 | 迁移方案 |
|-----|-----------|-----------|---------|
| 密码加密 | 无表结构变更 | 需迁移现有密码 | 编写迁移脚本，一次性加密 |
| 文章搜索 | 添加FULLTEXT索引 | 无破坏性变更 | 直接添加索引 |
| 评论功能 | 确认article_comment表 | 可能需补字段 | ALTER TABLE添加缺失字段 |
| 文章编辑/删除 | 无表结构变更 | 无影响 | 直接开发 |
| 图片上传 | 可选新增article_image表 | 无影响 | 新建表 |
| 草稿箱 | article表新增status字段 | 需设默认值 | ALTER TABLE添加字段，默认值1 |
| 用户关注 | 新增user_follow表 | 无影响 | 新建表 |
| 消息通知 | 新增notification表 | 无影响 | 新建表 |
| 数据统计 | article表新增view_count | 无影响 | ALTER TABLE添加字段，默认值0 |
| 标签升级 | 新增tag和article_tag表 | 需迁移数据 | 先建新表，后迁移article_sign数据 |

| 响应格式 | 保持现有的Result统一响应格式 |
| 认证方式 | 继续使用JWT Token认证 |

### 5.3 前端兼容性

| 原则 | 说明 |
|-----|------|

| 路由扩展 | 在现有嵌套路由结构上添加新路由 |
| 状态管理 | 新增Pinia store管理新功能的状态 |
| 样式统一 | 保持现有的毛玻璃+渐变风格 |

## 八、总结

本方案基于对当前个人博客系统的全面分析，从用户需求出发，结合技术可行性和系统兼容性，提出了三个阶段共12项功能丰富建议：

1. **第一阶段**聚焦核心功能补全，优先解决安全隐患（密码加密）和核心缺失功能（搜索、评论、文章编辑删除），预计2周完成。

2. **第二阶段**聚焦体验优化，补充图片上传、草稿箱、用户关注、消息通知等提升用户体验的功能，预计3周完成。

3. **第三阶段**聚焦高级增强，引入数据统计、标签系统升级、第三方登录、Redis缓存等提升系统竞争力的功能，预计3周完成。

建议按照优先级逐步实施，每个阶段完成后进行充分测试再进入下一阶段，确保系统的稳定性和用户体验的持续提升。
